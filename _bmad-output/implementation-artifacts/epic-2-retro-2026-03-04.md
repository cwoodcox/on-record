## Challenges and Growth Areas

### 1. API Integration Without Documentation (Critical)

**What happened:**
Both external APIs (UGRC GIS and Utah Legislature) were developed against guessed shapes. Story 2.1 used wrong district table names (`political.utah_house_districts` → `political.house_districts_2022_to_2032`) and wrong geometry format (`point:x,y` → ArcGIS JSON `point:{"x":x,"y":y}`). These same wrong values propagated into Story 2.4. Four bugs were discovered only during live testing and required two post-review bug-fix commits (3219e72, 27ab494).

**Root cause:**
No API documentation was obtained before create-story. The create-story process had no forcing function to require docs as an input.

**Fix:**
API docs are now required reading in the create-story invocation context for any story touching a third-party API. For Epic 3: Utah Legislature bills API docs are in hand before Story 3.1 is created.

---

### 2. Parallel Create-Story Spawned Dependent Specs Simultaneously (Critical)

**What happened:**
All six Epic 2 create-story agents were spawned in parallel. Story 2.4's spec was written at the same time as Story 2.1's spec — before `gis.ts` existed. The 2.4 spec described implementing a full inline `ugrcGeocode()` function, because the author had no way to know `resolveAddressToDistricts` in `gis.ts` would be created by 2.1.

When Story 2.4 was developed, `gis.ts` existed — but the story spec said to reimplement it. The dev agent followed the spec. The duplication was baked in from day one.

**Root cause:**
MEMORY.md said "create-story agents for sequential stories can run in parallel" without distinguishing independent stories from stories that produce interfaces or functions consumed by later stories. The guidance was correct for some cases and wrong for others.

**Fix:**
Update MEMORY.md: parallel create-story is safe only when stories don't produce interfaces/functions consumed by later stories in the same batch. Dependent stories must run create-story sequentially (after the producing story's spec documents what it creates).

For Epic 3 sequencing: 3.1 → 3.2 → [3.3 + 3.4 parallel] → 3.5; 3.6 parallel-safe from 3.2 onward.

---

### 3. Recurring Weak Test Assertions (Medium)

**What happened:**
All 6 code reviews caught missing or insufficiently specific test assertions. Most were low-stakes (missing aria-label tests, unchecked task checkboxes, wrong counts in completion notes). One was HIGH severity: Story 2.5's tool was forwarding internal error messages to users instead of user-friendly messages, masked by a test that checked `typeof result.nature === 'string'` instead of the actual value.

**Root cause:**
Dev agents write tests that prove code runs, not that it's correct. Type checks pass when the specific value is wrong.

**Fix:**
Add to code review checklist: error-path tests must assert specific `nature` and `action` string values, not type-only checks. If the exact message matters to users (and in three-field AppErrors it does), the test must pin it.

---

### 4. CI Currently Failing on Typecheck

**What happened:**
The post-Epic-2 bug fix to `tools/legislator-lookup.ts` (commit 27ab494 — correcting API table names, geometry format, and input schema) introduced TypeScript errors. CI typecheck is currently failing in `apps/mcp-server/`.

**Root cause:**
The inline `ugrcGeocode()` function was patched rather than removed. The correct fix is to remove the duplicate GIS logic entirely and import `resolveAddressToDistricts` from `gis.ts` as originally intended by Story 2.1.

**Fix (critical path item):**
Remove inline GIS reimplementation from `tools/legislator-lookup.ts`. Import `resolveAddressToDistricts` from `../lib/gis.js`. Restore green CI. This must be done before Epic 3 starts.

## Key Insights

1. **The process failure was create-story sequencing, not dev agent behavior.** Dev agents followed their specs faithfully. The specs were wrong because they were written without the context of what parallel stories would create.

2. **Mocks-only testing can pass 100% green while being wrong about the exact things that matter.** Unit tests validated our guessed API shapes against themselves. The only signal that mattered came from live integration testing.

3. **The MEMORY.md guidance on parallel create-story is incomplete.** One sentence of clarification would have prevented the Story 2.4 duplication entirely.

4. **API docs are a project input, not a dev-time discovery.** Both API issues (Epic 2) would have been non-issues if docs were read during create-story rather than discovered during live testing.
