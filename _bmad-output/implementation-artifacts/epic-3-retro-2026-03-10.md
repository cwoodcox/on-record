## Challenges

1. **Story 3.2 required 3 code review rounds** — Most complex story in the epic. `writeBills` edge cases (mixed-session handling, overwrite-before-insert), scheduler wiring untested until second review, FTS5 rebuild correctness. Complexity was underestimated in planning.

2. **`getActiveSession()` stub chain across three stories** — Stubbed in 3.2 (3.4 hadn't been built), ripped out and replaced in 3.4, wrapped again with a singleton accessor in 3.5. Three stories touched session detection logic. Dependency not fully designed upfront.

3. **Weak test assertions recurring despite checklist (4 of 6 reviews)** — Tests prove code runs, not that it returns the correct value. `toHaveBeenCalledWith` and pinned string assertions frequently missing. The checklist corrects but doesn't prevent — spec-level requirements are the forcing function, not post-write review.

4. **API behavior surprises — 200 OK non-JSON body, session structure** — Utah Legislature API returns HTTP 200 on failure with a non-JSON body (HTML or error string). Session structure didn't match assumptions. Both caught and fixed within their respective stories (3.1, 3.4), but still caused rework. No post-review commits needed.

5. **Tool description gated LLM behavior** — The `search_bills` tool description enumerated the supported theme categories. Copilot + GPT read this as a constraint and presented a category menu instead of accepting freeform input — blocking the fallback path. The fallback does work when bypassed (confirmed: "other → voting rights" returns results).

6. **Testing environment validity** — Testing with Claude/Copilot instances that participated in code reviews introduces implementation memory bias. These instances knew about "the bill cache," FTS5 categories, retry logic — things a real user's LLM wouldn't know. Current UX testing is partially unreliable as a result.

---

## Key Insights

1. **Process checklist ≠ prevention** — A checklist corrects after the fact. The right place to enforce quality is in acceptance criteria, where the dev agent writes to spec from day one.

2. **Utah Legislature API is a temporary shim** — Post-MVP, OpenStates replaces the provider layer entirely. Better docs, 50-state coverage, structured vote data. Don't over-invest in Utah Legislature API process; the migration will be clean because of the boundary discipline already in place (`providers/utah-legislature.ts` isolated, cache layer portable).

3. **API discovery before spec-writing is a project standard** — Calibrated to doc quality: if docs are sparse, do a discovery spike to fill gaps and spot-check error cases (including triggering documented error responses). If docs are comprehensive with error examples, they can be trusted.

4. **Tool descriptions shape LLM behavior** — The schema and description need to be designed with the mediating LLM in mind. Enumerating valid values implies they're the only valid values. Describing intent (freeform search term derived from constituent's concern) is safer than describing implementation (list of categories).

5. **Sponsored bills only — known scope boundary** — The current implementation surfaces sponsored legislation, not full voting record. The Utah Legislature API doesn't expose vote records in a structured queryable way. The Epic 4 system prompt must represent this accurately; the LLM must not imply it can surface voting record when it cannot.

6. **Testing isolation matters** — System prompt behavior tests must use clean LLM sessions with no implementation memory. Results from sessions that participated in development are not reliable signals of real user experience.
